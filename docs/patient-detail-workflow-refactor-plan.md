# 患者详情页流程任务驱动重构实施方案

## 1. 背景与目标

当前系统的核心业务对象是“患者健康管理档案”。档案中约 136 个字段基本都来自实际业务需要，不能简单删除或弱化。现有系统已经具备字段分级、分权限填写、查看、编辑、医生复核、归档等基础流程闭环。

但随着字段数量和协作角色增多，当前患者详情页逐渐暴露出几个问题：

1. 字段数量多，管理员视角容易被信息淹没。
2. 部分字段需要多岗位协作填写、补充、查看和复核，不能简单固定为某个岗位独占。
3. 当前页面更像“字段表单集合”，而真实业务更像“流程节点任务协作”。
4. 岗位字段分配越细，越容易出现职责混乱、重复填写、机械敷衍。
5. 患者详情页作为核心功能展示点，承担了登记、接诊、检查、治疗、缴费、入院、出院、归档、随访等过多职责。

本方案目标不是删除字段，也不是推翻现有业务，而是在保持核心功能不变的前提下，将患者详情页从“字段驱动”逐步重构为“流程任务驱动”。

最终希望达到：

- 字段完整性保留。
- 岗位职责更清晰。
- 管理员默认看到流程完整性和阻塞项。
- 各岗位默认看到当前节点需要处理的任务。
- 协作字段明确区分“主责、协作、复核、查看”。
- 患者详情页视觉上更轻、更聚焦、更适合生产使用。

---

## 2. 核心设计原则

### 2.1 不再以“岗位字段”作为唯一组织方式

现有系统已经有分岗位字段填写和查看能力，但真实业务中很多字段并不是某个岗位独占，而是存在：

- 首填岗位
- 主责岗位
- 协作岗位
- 复核岗位
- 查看岗位

因此后续字段管理不应继续只问“这个字段归哪个岗位填”，而应改为：

> 这个字段在哪个流程节点产生？谁首填？谁主责？谁可以补充？谁必须复核？谁需要查看？

### 2.2 从字段驱动转为流程任务驱动

当前页面的核心问题不是字段本身太多，而是用户进入页面后直接面对大量字段。

后续页面应优先回答：

- 当前病人走到哪个流程节点？
- 当前节点谁负责？
- 当前还缺什么？
- 哪些内容需要复核？
- 哪些附件异常？
- 是否可以进入下一步？
- 是否可以归档？

字段是任务背后的数据，不应成为第一屏的全部内容。

### 2.3 保留完整档案，但降低默认暴露度

136 个字段仍然保留在完整档案中。

但默认视图应优先展示：

- 患者摘要
- 当前流程节点
- 待处理任务
- 阻塞项
- 风险提示
- 附件状态
- 医生复核状态
- 质控归档状态

完整字段放在“完整档案 / 高级查看 / 全部字段 / 打印预览”中。

### 2.4 先做信息架构，再做代码重构

不建议直接大规模重写页面。

推荐顺序：

1. 明确字段责任模型。
2. 明确流程节点任务模型。
3. 调整 UI 信息架构。
4. 再逐步拆组件和 composable。
5. 最后再做智能质控和低质量填写检测。

## 3. 业务流程抽象

根据当前实际业务，患者完整流程可抽象为 8 个节点。

```text
1. 前台登记
2. 接诊 / 院长初诊沟通
3. 检查室详细检查
4. 接诊 / 院长确认治疗路径
5. 缴费
6. 医师治疗 / 护理部入院办理
7. 出院 / 治疗结束
8. 归档 / 质控 / 后续随访
```

后续所有字段、权限、页面任务，都应挂靠到这些流程节点下。

## 4. 流程节点职责设计

### 4.1 前台登记

主责岗位：前台  
协作岗位：接诊室、管理员  
复核岗位：接诊室或管理员

主要目标：建立患者基础档案，确保患者身份和就诊入口信息准确。

字段类型：

- 患者姓名
- 性别
- 年龄
- 联系方式
- 就诊号
- 到院时间
- 来源渠道
- 初诊 / 复诊
- 陪同人员
- 初步诉求
- 基础备注

设计建议：

- 前台只负责事实性基础信息。
- 前台不承担过多临床判断。
- 接诊室可以补充患者诉求。
- 管理员可以复核基础资料完整性。

---

### 4.2 接诊 / 院长初诊沟通

主责岗位：接诊室 / 院长  
协作岗位：前台、检查室  
复核岗位：院长 / 医生

主要目标：了解病情、患者诉求和沟通重点，形成初步判断。

字段类型：

- 主诉
- 现病史
- 既往史
- 疼痛情况
- 患者重点诉求
- 家属意见
- 沟通重点
- 初步判断
- 是否建议检查
- 是否建议门诊治疗
- 是否建议入院
- 患者顾虑

设计建议：

- 接诊室可以首填。
- 院长可以复核、补充和确认。
- 检查室可以查看与检查相关的病情背景。
- 前台只查看必要结论，不参与临床判断。

---

### 4.3 检查室详细检查

主责岗位：检查室  
协作岗位：医生 / 院长  
复核岗位：医生 / 院长

主要目标：完成检查事实记录，上传检查附件，为治疗路径判断提供依据。

字段类型：

- 检查项目
- 检查结果
- 检查描述
- 检查图片 / 附件
- 异常发现
- 检查建议
- 检查完成状态
- 是否需补查

设计建议：

- 检查室负责填写事实和上传附件。
- 医生 / 院长负责复核临床意义。
- 附件状态应明确区分：已上传、待确认、异常、已作废。
- 检查结果和医生判断不应混为一个字段。

---

### 4.4 确认治疗路径

主责岗位：院长 / 接诊室  
协作岗位：检查室、医生、前台  
复核岗位：院长

主要目标：基于病情和检查结果，确定门诊治疗或入院路径。

字段类型：

- 最终诊断
- 治疗方式
- 门诊治疗 / 入院判断
- 治疗风险
- 沟通结果
- 患者是否同意
- 是否需缴费
- 是否需入院手续

---

## 组件拆分样式约定

拆分长页面时，子组件必须持有自身 DOM class 对应的 scoped 样式。父页面只保留页面级布局容器、主题变量和跨组件间距，不再依赖父 SFC 的 scoped 选择器去影响子组件内部结构。

这样可以避免 `.vue` scoped 编译后，原本写在父页面里的选择器无法命中新抽子组件 DOM，导致按钮、弹窗、表格和上传面板样式丢失。后续每次抽组件都按“模板一起搬、局部样式一起搬、业务状态留父页面”的顺序执行。

- 下一步安排

设计建议：

- 这是关键决策节点。
- 字段应有状态：草稿、已确认、已归档。
- 接诊室可以填写草稿。
- 院长确认后进入后续流程。
- 前台、护理、医生根据确认结果执行后续动作。

---

### 4.5 缴费

主责岗位：前台 / 收费  
协作岗位：接诊室  
复核岗位：管理员

主要目标：确认费用状态，判断是否允许进入治疗或入院办理。

字段类型：

- 缴费状态
- 缴费项目
- 缴费时间
- 欠费 / 补费说明
- 是否允许进入治疗

设计建议：

- 缴费更偏流程状态，不应和医疗档案字段深度混杂。
- 健康档案中只保留与流程推进相关的摘要信息。
- 详细收费记录可以作为外部流程或独立模块。

---

### 4.6 医师治疗 / 护理部入院办理

该节点存在两条路径。

#### 4.6.1 门诊治疗路径

主责岗位：医师  
协作岗位：护理、检查室  
复核岗位：院长 / 质控

字段类型：

- 治疗项目
- 治疗时间
- 操作医生
- 治疗记录
- 术中情况
- 用药情况
- 不良反应
- 医嘱
- 复查建议

设计建议：

- 医师负责治疗事实和医学判断。
- 护理可以补充护理配合信息。
- 院长或质控复核关键治疗记录。

#### 4.6.2 入院路径

主责岗位：护理部  
协作岗位：医生、前台  
复核岗位：医生 / 管理员

字段类型：

- 入院手续状态
- 床位 / 护理信息
- 入院宣教
- 护理注意事项
- 治疗前准备
- 风险告知
- 入院后治疗衔接

设计建议：

- 入院办理和治疗记录应区分。
- 护理部负责手续与护理准备。
- 医生负责治疗衔接和医学判断。

---

### 4.7 出院 / 治疗结束

主责岗位：前台 / 护理 / 医师  
协作岗位：管理员  
复核岗位：医生 / 质控

主要目标：完成治疗或出院流程，生成后续复查和随访安排。

字段类型：

- 治疗结束时间
- 出院状态
- 患者离院情况
- 费用是否结清
- 复查时间
- 注意事项
- 患者反馈
- 满意度
- 后续随访计划

设计建议：

- 前台可处理流程结束。
- 医疗结论和复查建议应由医生确认。
- 管理员关注是否满足归档条件。

---

### 4.8 归档 / 质控 / 后续随访

主责岗位：管理员 / 质控  
协作岗位：所有岗位  
复核岗位：院长 / 医生

主要目标：确认档案完整、可追溯、可归档，并形成后续随访闭环。

字段类型：

- 档案完整度
- 缺失字段
- 附件异常
- 医生复核状态
- 质控意见
- 退回原因
- 归档时间
- 随访状态
- 下次复查
- 问题闭环

设计建议：

- 管理员不应默认面对全部字段。
- 管理员应默认看到流程完整性、阻塞项和异常项。
- 质控视图应以问题清单为核心，而不是字段表单。

---

## 5. 字段分类模型

建议将 136 个字段重新映射为 4 类。

### 5.1 事实类字段

谁最早获得事实，谁填写。

示例：

- 患者姓名
- 到院时间
- 检查结果
- 缴费状态
- 附件上传

特点：

- 一线岗位可填写。
- 结构化程度高。
- 不需要太多主观判断。

### 5.2 判断类字段

必须由专业岗位确认。

示例：

- 诊断
- 治疗方案
- 是否入院
- 风险判断
- 医嘱
- 复查建议

特点：

- 可由协作岗位补充。
- 最终必须由医生 / 院长确认。
- 应具备复核状态。

### 5.3 过程类字段

记录流程推进状态。

示例：

- 当前节点
- 是否检查完成
- 是否缴费
- 是否治疗
- 是否出院
- 是否归档

特点：

- 更适合做时间轴和流程状态。
- 不应和医疗字段平铺混在一起。

### 5.4 质控类字段

管理员和质控重点关注。

示例：

- 必填缺失
- 附件异常
- 填写质量
- 复核状态
- 退回意见
- 归档意见

特点：

- 管理员主责。
- 应以问题清单形式展示。
- 不一定需要所有岗位看到。

## 6. 字段责任表设计

建议为 136 个字段建立字段责任表。

每个字段至少补充以下信息：

```text
字段名
所属流程节点
字段类型：事实 / 判断 / 过程 / 质控
首填岗位
主责岗位
协作岗位
复核岗位
查看岗位
是否归档必填
是否关键字段
是否允许追加
是否允许覆盖
是否需要附件证据
是否需要低质量填写检测
```

示例：

```text
字段：最终治疗方案
所属节点：确认治疗路径
字段类型：判断
首填岗位：接诊室 / 院长
主责岗位：院长
协作岗位：医生、检查室
复核岗位：院长
查看岗位：前台、护理、医生、管理员
是否归档必填：是
是否关键字段：是
是否允许追加：是
是否允许覆盖：需权限
```

```text
字段：检查附件
所属节点：检查室详细检查
字段类型：事实 / 证据
首填岗位：检查室
主责岗位：检查室
协作岗位：医生
复核岗位：医生 / 管理员
查看岗位：接诊室、院长、医生、管理员
是否归档必填：视检查项目而定
是否关键字段：是
是否允许追加：是
是否允许覆盖：不建议覆盖，采用作废 + 新增
```

## 7. 页面信息架构建议

### 7.1 患者详情页目标结构

建议将患者详情页调整为“健康档案工作台”。

```text
患者详情页
├─ 顶部：患者身份 + 当前阶段 + 风险 + 主操作
├─ 左侧：流程 / 场景导航
│  ├─ 总览
│  ├─ 待处理
│  ├─ 前台登记
│  ├─ 接诊沟通
│  ├─ 检查附件
│  ├─ 治疗路径
│  ├─ 治疗 / 入院
│  ├─ 出院 / 结束
│  ├─ 质控归档
│  └─ 完整档案
├─ 中间：当前任务或字段内容
│  ├─ 当前节点任务
│  ├─ 阻塞项
│  ├─ 关键字段
│  ├─ 协作字段
│  └─ 折叠字段组
└─ 右侧：上下文辅助栏
   ├─ 最近更新
   ├─ 附件证据
   ├─ AI 总结
   ├─ 时间轴
   └─ 下一步建议
```

### 7.2 管理员默认视图

管理员默认不进入完整字段表单。

管理员默认看到：

```text
质控总览
├─ 流程进度
├─ 当前阻塞项
├─ 必填缺失
├─ 附件异常
├─ 医生待复核
├─ 最近修改
├─ 可归档判断
└─ 下一步建议
```

流程进度示例：

```text
前台登记：已完成
接诊沟通：待院长确认
检查室：附件缺 1 项
治疗路径：待确认
缴费：已完成
治疗：进行中
出院：未开始
归档：不可归档
```

### 7.3 岗位默认视图

岗位默认不再叫“我的字段”，而是叫：

```text
当前节点任务
```

每个岗位看到：

```text
我负责填写
我可以补充
我需要复核
我只能查看
```

这样可以明确协作字段的处理关系。

## 8. UI 优化方向

### 8.1 默认阅读态，按需编辑

136 个字段不应全部以输入框形态展示。

建议：

- 已填写字段默认以阅读卡片展示。
- 点击后进入编辑态。
- 缺失字段以“待补充”提示展示。
- 异常字段以风险样式展示。

这样可以显著降低视觉噪音。

### 8.2 字段权重分级

字段视觉权重建议分为：

| 字段级别     | 展示方式                   |
| ------------ | -------------------------- |
| 核心字段     | 第一屏展示，大字号或摘要卡 |
| 关键复核字段 | 蓝色 / 黄色状态提示        |
| 普通记录字段 | 折叠展示                   |
| 低频字段     | 默认隐藏，搜索可见         |
| 历史追溯字段 | 时间轴或完整档案中展示     |

### 8.3 增加“只看待处理”模式

该模式只展示：

- 必填缺失
- 格式异常
- 附件缺失
- 医生待确认
- 管理员待复核
- 冲突草稿
- 保存失败

这是生产环境中最实用的模式之一。

### 8.4 增加字段搜索和快速定位

顶部增加搜索入口：

```text
搜索字段、附件、复查节点
```

搜索结果直接定位字段并高亮。

### 8.5 收敛顶部按钮

当前顶部动作较多，建议收敛为：

- 主操作：保存 / 提交质控 / 打印
- 更多操作：AI 总结、患者助手、操作轨迹、复制、导入旧资料

减少第一屏按钮压力。

## 9. 代码层实施建议

### 9.1 第一阶段：新增总览和任务模型

新增组件建议：

- `PatientOverviewPanel.vue`
- `WorkflowProgressPanel.vue`
- `QualityIssuePanel.vue`

## 11. 验证方式

每一阶段都应验证：

1. 前台登记流程是否正常。
2. 接诊 / 院长是否能查看和补充病情信息。
3. 检查室是否能上传和查看附件。
4. 医生是否能看到检查结果并确认治疗方案。
5. 护理 / 入院流程是否不受影响。
6. 前台是否能办理结束 / 出院相关状态。
7. 管理员是否能看到完整档案、问题清单和归档状态。
8. 打印 / PDF 是否正常。
9. AI 总结、患者助手、时间轴、操作轨迹是否正常。
10. `type:check`、`lint:prettier:check`、`lint:eslint:check`、`build:pro` 是否通过。

## 12. 最终目标形态

最终患者详情页应从：

```text
一个包含 136 个字段的大表单页面
```

逐步变成：

```text
围绕患者健康管理流程运转的协作工作台
```

最终用户进入页面后应优先知道：

- 患者现在处于哪个节点。
- 当前谁需要处理。
- 哪些信息缺失。
- 哪些字段需要医生复核。
- 哪些附件异常。
- 是否可以进入下一步。
- 是否可以归档。

完整字段仍然存在，但不再主导第一屏体验。
