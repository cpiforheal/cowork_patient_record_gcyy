# 后端上线排障与可观测性清单

本文用于线上或内网试运行时快速定位“不可预知的问题”。本轮只记录后端工程化信号，不改变业务接口。

## 请求定位

- 每个非静态资源请求都会返回响应头 `X-Request-Id`。
- 后端访问日志会记录同一个 `requestId`、请求方法、路径、状态码、耗时、登录用户和来源 IP。
- 500 类错误返回给前端的提示中会带同一个 `requestId`，用户截图后可直接按该编号查日志。
- 不记录请求体、token、AI key、数据库密码等敏感内容。

## 健康检查

公开地址：

```text
GET /health
```

返回服务状态、启动时间、运行时长和组件摘要：

- `database`：数据库是否可连通。
- `attachmentStorage`：附件目录是否可写、剩余空间。
- `backup`：备份是否启用、是否已配置目录、最近一次备份状态。
- `ai`：AI provider 是否已配置，不暴露密钥。

登录后地址：

```text
GET /health/db
```

保留数据库详细检查，包含数据库类型、版本和连通耗时。该接口继续走登录态校验。

## 启动自检

服务启动时会检查：

- 当前激活 profile。
- 后端端口。
- MySQL 目标地址和数据库账号，日志不输出密码。
- 附件目录是否可创建和写入。
- 备份定时表达式。
- AI、豆包、TTS 是否完成必要配置，日志只输出布尔状态。

如果附件目录不可写，后端会启动失败。先修复 `CLINIC_ATTACHMENT_DIR` 路径、权限或磁盘空间，再重新启动。

## 常用定位顺序

1. 浏览器或前端弹窗里记录 `X-Request-Id`。
2. 查看 `logs/backend.out.log` 和 `logs/backend.err.log`，搜索该 requestId。
3. 访问 `/health` 判断是数据库、附件目录、备份还是 AI 配置问题。
4. 若数据库异常，用管理员账号访问 `/health/db` 查看详细连通信息。
5. 上传或附件展示异常时，优先检查 `CLINIC_ATTACHMENT_DIR` 权限和磁盘剩余空间。

## 配置文件

真实运行配置放在：

```text
config/runtime.env
```

模板文件是：

```text
config/runtime.env.example
```

真实密码、AI key、生产环境路径不要提交到 git。
