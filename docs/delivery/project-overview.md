# 项目介绍与内测边界

## 一句话定位

本系统是面向内网门诊一线工作的“患者病历协同 + 科室物资申领与库存追溯台账”平台，用于替代分散的纸质记录、Excel 台账和口头流转，让患者资料、附件、物资申领、发放、签收、消耗、盘点和追溯尽量集中到同一个可查询、可复核的工作台中。

## 当前主要解决的问题

- 病历资料分散：患者登记、门诊记录、附件资料和历史记录集中维护。
- 多岗位协作不清：前台、医生、护士、检查/检验、质控等岗位在同一患者流程中协同。
- 物资申领不可追溯：科室申请、负责人审核、库管发放、科室签收、周消耗、盘点差异和报废记录形成台账。
- 领导检查成本高：通过主控台、领导驾驶舱、风险清单和导出数据，快速判断哪些事项未闭环。
- 内测部署门槛高：交付包提供检查、启动、停止脚本，减少非开发人员操作成本。

## 当前模块范围

### 患者病历协同

- 今日患者与患者流程查看。
- 患者基础信息维护。
- 门诊/住院相关病历字段维护。
- 附件上传、查看、作废与恢复。
- 病历模板、质控审核、操作记录和数据留痕。

### 进销存管理

侧边栏统一收束在“进销存管理”下，内含：

- 进销存主控台：查看待办、风险、闭环状态和常用入口。
- 领导驾驶舱：查看关键指标、趋势和异常事项。
- 申领审批：科室申领、负责人审核、库管发放、科室签收。
- 库存看板：查看库存、批次、低库存、临期和过期情况。
- 物资档案：维护物资名称、规格、单位、预警线、批号/效期要求。
- 周消耗：记录实际消耗、剩余数量和下周预计领用。
- 盘点处置：处理盘点差异、退回和报废。
- 追溯流水：查询库存变化、操作人、时间和原因。

## 第一轮内测边界

第一轮内测的目标不是一次性替代完整 HIS、财务 ERP 或采购系统，而是验证下面四件事：

1. 一线岗位是否愿意用系统完成日常登记、填写、上传、申领和签收。
2. 科室负责人和库管是否能通过系统完成审核、发放、盘点和追溯。
3. 领导是否能通过看板和导出数据判断“今天有没有事、谁还没闭环、库存是否异常”。
4. 系统在固定内网主机、MySQL 数据库和附件目录下能否稳定运行、可备份、可恢复。

## 内测期间的管理约定

- 内测账号只发给实际参与人员，不借用、不共用。
- 患者资料和附件只在单位内网环境使用，不复制到个人电脑长期保存。
- MySQL 数据库和附件目录必须一起备份，缺少任意一项都不能完整恢复业务资料。
- 物资档案先由管理人员统一整理，再开放科室申领，避免名称、规格和单位混乱。
- 内测问题优先按“是否影响闭环”记录：不能保存、不能签收、不能发放、数据找不到、附件打不开属于优先处理事项。

## 不在第一轮内测强行覆盖的事项

- 与外部 HIS、医保、财务、采购平台的正式接口对接。
- 扫码枪、移动端深度适配、供应商采购单和复杂财务核算。
- 多院区、多实例部署和跨公网访问。
- 自动月报、PDF 周报、复杂权限配置后台等增强能力。

这些能力可以在第一轮内测稳定后继续扩展。第一轮更重要的是让系统先在真实岗位中跑通，并证明它比纸质和 Excel 更省事、更容易追溯。
