# 内测复核与验收清单

本文用于第一轮内测前的最终复核。目标是确认系统能在固定边界内完成业务闭环，而不是追求所有扩展能力一次到位。

## 项目复核结论

当前系统原型已具备第一轮内测所需的主要闭环：

- 患者病历资料可登记、填写、保存、查看和上传附件。
- 门诊常用岗位可以围绕同一患者资料协同。
- 进销存已收束在独立菜单下，覆盖物资建档、入库、申领、审核、发放、签收、周消耗、盘点、退回、报废和流水追溯。
- 交付包已提供检查、启动、停止和配置入口，适合非开发人员在说明文档指导下运行。
- 领导和管理人员可以通过主控台、驾驶舱、风险清单、库存看板和流水导出进行内测检查。

## 上线前必走验收

### 基础访问

- [ ] 目标主机固定 IP 已确认。
- [ ] `01-check-host.bat` 检查通过。
- [ ] 本机可访问系统首页。
- [ ] 至少一台其他内网电脑可访问系统首页。
- [ ] 登录、退出、重新登录正常。

### 病历协同

- [ ] 新建一个测试患者。
- [ ] 填写并保存患者基础信息。
- [ ] 填写一段门诊病历或相关记录。
- [ ] 上传一张图片附件。
- [ ] 上传一个 PDF 附件。
- [ ] 关闭浏览器后重新进入，患者资料仍可查看。
- [ ] 重启系统后重新进入，患者资料和附件仍可查看。
- [ ] 作废或恢复测试附件后，记录能被查看。

### 进销存闭环

- [ ] 新建一个测试物资档案。
- [ ] 完成一次入库。
- [ ] 科室提交一次申领。
- [ ] 负责人完成审核。
- [ ] 库管完成发放。
- [ ] 科室完成签收。
- [ ] 库存看板数量同步变化。
- [ ] 追溯流水能查到对应操作。
- [ ] 填写一次周消耗。
- [ ] 完成一次盘点记录。
- [ ] 导出一次库存或流水 CSV。

### 数据与备份

- [ ] 已确认 MySQL 数据库名称和账号。
- [ ] 已确认附件目录绝对路径。
- [ ] 已完成一次数据库备份演练。
- [ ] 已复制一次附件目录备份。
- [ ] 已保存当前交付包和 `runtime.env`。
- [ ] 已记录系统负责人、数据库负责人、附件备份负责人。

## 内测期间重点观察

### 一线岗位

- 是否能不依赖开发人员完成登录和日常录入。
- 是否愿意在系统内完成申领、签收和周消耗。
- 是否存在字段过多、找不到入口、保存后不确定是否成功等问题。

### 科室负责人

- 是否能每天从待办中找到需要审核的事项。
- 是否能判断哪些申领合理、哪些需要退回线下沟通。
- 是否能追踪本科室长期未签收或异常消耗。

### 物资管理员

- 是否能用系统替代主要 Excel 台账。
- 是否能准确处理入库、发放、盘点、退回和报废。
- 是否能在领导要求时快速查到某个物资的历史流水。

### 领导和管理人员

- 是否能通过驾驶舱和主控台判断今日状态。
- 是否能看到未闭环事项。
- 是否能导出数据用于会议、检查或留档。

## 内测问题记录模板

建议每个问题按下面格式登记：

| 字段 | 内容 |
| --- | --- |
| 发生时间 | 例如 2026-06-17 09:30 |
| 科室/岗位 | 例如 内科/护士 |
| 操作入口 | 例如 进销存管理 -> 申领审批 |
| 问题描述 | 简明写清发生了什么 |
| 是否影响闭环 | 是/否 |
| 截图或附件 | 文件名或保存位置 |
| 临时处理方式 | 例如 线下补登、重新提交、管理员处理 |
| 后续结论 | 修复、培训、规则调整或暂不处理 |

## 内测收敛标准

建议满足下面条件后，再扩大科室范围：

- 连续 5 个工作日系统可正常访问。
- 测试科室能完成患者资料和物资申领闭环。
- 关键附件上传和查看正常。
- 库存数量、流水和导出数据能被库管接受。
- 领导能通过看板发现未闭环事项。
- 已形成固定备份动作。
- 已明确内测问题反馈人和处理节奏。
